The token-level F1 is misleading because 85% of the tokens are `O`, so a model can look strong on token F1 just by getting the majority non-entity class right. The span-level seqeval number is the honest entity extraction metric here, and `0.61` suggests the model is still making too many entity-level mistakes to call it good.

Mechanically, the gap means the model is probably getting many entities roughly in the right place but missing exact boundaries. Seqeval gives credit only on an exact span match, so the full span must match exactly. If the model starts a span one token late, ends it one token early, or over-extends into a title token, the whole span is wrong even though only one or two BIO tags were off. That kind of boundary drift barely dents token F1 but crushes span F1.

To diagnose the dominant error type, I would compute span precision and recall separately, break down BIO transition errors such as `I-CHAR` without a preceding `B-CHAR`, and compare exact-match span F1 to a partial-overlap metric. I would also count truncation versus extension errors on multi-token names and titles in fantasy fiction, because things like `the First Enchantress` are much more likely to boundary-drift than single-token names.
